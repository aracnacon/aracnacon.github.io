What Is OpenTelemetry?

OpenTelemetry (OTel) is a vendor-neutral, open-source observability framework governed by the Cloud Native Computing Foundation (CNCF). It provides a single standard for collecting three core signals across any distributed system:

  • Metrics — Numeric measurements over time (CPU usage, latency, error rates, throughput)
  • Traces — End-to-end request flows across services, composed of spans with timing and attributes
  • Logs — Structured or unstructured event records, bridged from existing loggers into the OTel pipeline

The core principle: instrument once, export anywhere. No vendor lock-in. One SDK, any backend.

Architecture Overview

Every OTel deployment follows the same pattern — applications emit telemetry via SDKs, which flows through a central Collector that routes data to your chosen backends:

  • Receivers — Accept data from apps via OTLP, Prometheus, Jaeger, or Zipkin protocols
  • Processors — Transform, filter, batch, and limit memory usage before export
  • Exporters — Send processed data to any backend (Prometheus, Jaeger, Datadog, Loki, etc.)
  • Extensions — Add-ons like health checks, pprof profiling, and zPages diagnostics

Multi-Language Instrumentation

OTel provides SDKs and auto-instrumentation for every major language. Each follows the same pattern — configure a resource, set up trace and metric exporters, point at your Collector, and start the SDK:

  • Node.js — @opentelemetry/sdk-node with auto-instrumentations for Express, HTTP, and database clients. Load tracing.js before your app entry point via node -r ./tracing.js app.js
  • Python — opentelemetry-distro with zero-code auto-instrumentation CLI: opentelemetry-instrument --service_name my-service python app.py
  • Go — go.opentelemetry.io/otel with manual TracerProvider setup and OTLP HTTP exporter
  • Java — Spring Boot starter or zero-code Java agent: java -javaagent:opentelemetry-javaagent.jar -jar app.jar
  • .NET — OpenTelemetry.Extensions.Hosting with ASP.NET Core and HttpClient instrumentation wired in Program.cs

Every SDK respects standard environment variables (OTEL_SERVICE_NAME, OTEL_EXPORTER_OTLP_ENDPOINT, OTEL_TRACES_SAMPLER) for zero-code configuration.

Hooks & Lifecycle

OTel hooks let you intercept pipeline stages to add custom attributes, filter noise, and trigger side effects:

  • SpanProcessor (onStart / onEnd) — Inject custom attributes on span creation, log timing on span close, forward to custom sinks
  • HTTP Instrumentation Hooks — Enrich inbound/outbound HTTP spans with custom headers, content length, or route metadata. Ignore noise routes like /healthz
  • Observable Metric Callbacks — Register async callbacks for expensive metrics (CPU usage, queue depth) instead of polling every tick
  • Custom Samplers — Route-based or load-based sampling decisions: always capture errors and critical paths, sample 10% of everything else
  • SDK Lifecycle — Wire shutdown to SIGTERM/SIGINT to flush buffered telemetry before process exit. Prevents data loss on deploys

Lighthouse & Web Vitals

Bridging frontend performance into the OTel pipeline for unified full-stack observability:

  • Core Web Vitals → OTel Metrics — Capture LCP, INP, CLS, FCP, and TTFB in the browser using the web-vitals library, export as OTel histograms to the Collector
  • Lighthouse CI — Run Lighthouse in CI pipelines, push performance/accessibility/SEO scores as OTel metrics for trend tracking in Grafana
  • Alerting — Prometheus alerting rules on CWV thresholds: P75 LCP > 2.5s, CLS > 0.1, Lighthouse performance drops below 85%

Collector & Deployment

The OTel Collector is the central routing component that decouples your application from its backend. I deploy it as part of a full observability stack:

  • Docker Compose — Collector + Jaeger + Prometheus + Grafana in a single docker-compose.yml. OTLP ports (4317 gRPC, 4318 HTTP) exposed, Prometheus scrapes at :8889, Jaeger UI at :16686
  • Kubernetes (Helm) — OTel Operator via Helm chart, DaemonSet or Sidecar mode. Auto-instrumentation via pod annotations: instrumentation.opentelemetry.io/inject-python: "true"
  • Pipeline Config — YAML-driven pipelines: receivers (OTLP + Prometheus scrape) → processors (memory_limiter, batch, filter health checks) → exporters (Prometheus, Jaeger, Loki)
  • Security — TLS on Collector endpoints in production, auth headers for SaaS backends, sensitive data filtering in processors, health/debug endpoints blocked from public access

Exporter Compatibility

OTel's vendor-neutral design means you can swap backends without changing application code. If a backend supports OTLP natively, use the otlp exporter directly:

  • Prometheus — Metrics (exposes scrape endpoint)
  • Jaeger / Tempo — Traces (OTLP native)
  • Loki — Logs (Grafana's log backend)
  • ClickHouse — All signals at scale
  • Elasticsearch — Logs and trace search
  • Datadog / New Relic / Honeycomb / Dynatrace — SaaS backends via OTLP or named exporters

My Build Checklist

Every service I instrument follows this checklist to ensure complete, production-ready observability:

  • OTel SDK added with service.name, service.version, and environment resource attributes
  • Trace and metric exporters configured pointing to the Collector
  • Auto-instrumentation enabled for HTTP, database, and messaging libraries
  • Context propagation configured (tracecontext + baggage)
  • SDK shutdown wired to SIGTERM for graceful buffer flushing
  • Collector deployed with memory_limiter and batch processors
  • Health check routes filtered to reduce trace noise
  • TLS and auth headers configured for production endpoints
  • Dashboards created for golden signals (latency, errors, throughput, saturation)
  • Alerts configured on key metrics and CWV regressions
